## Codebase Patterns
- Expense lifecycle events use `record_expense_event()` with idempotency key `{seller}:{payment_id}:{event_type}` (3 parts)
- expense_captured uses sign "any" (positive=income, negative=expense), other expense events use "zero"
- All event_ledger queries (get_dre_summary, get_payment_statuses) already filter out `expense_*` events — same as `cash_*`
- Tests in `testes/test_event_ledger.py` cover pure functions; async tests use `unittest.mock.patch` on `record_event`
- Full test suite: 411 tests, ~2.2s
- Dual-write helper pattern: `_dual_write_expense_events()` as async helper, called after mp_expenses upsert, catches `EventRecordError` as warning
- Metadata for expense_captured must include ALL fields for Fase 3: expense_type, expense_direction, ca_category, auto_categorized, description, amount, date_created, date_approved, business_branch, operation_type, payment_method, external_reference, beneficiary_name, notes
- Signed amount: positive for income direction, negative for expense/transfer
